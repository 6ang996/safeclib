11 Overview
12 --------
13 The ISO TR24731 Bounds Checking Interface documents indicate that the key
14 motivation for the new specification is to help mitigate the ever increasing
15 security attacks, specifically the buffer overrun.
16 
17 The rationale document says ``Buffer overrun attacks continue to be a security
18 problem. Roughly 10% of vulnerability reports cataloged by CERT from
19 01/01/2005 to 07/01/2005 involved buffer overflows. Preventing buffer overruns
20 is the primary, but not the only, motivation for this technical report.''
21 
22 The rationale document continues ``that these only mitigate, that is lessen,
23 security problems. When used properly, these functions decrease the danger
24 buffer overrun attacks. Source code may remain vulnerable due to other bugs
25 and security issues. The highest level of security is achieved by building in
26 layers of security utilizing multiple strategies.''
27 
28 .The rationale document lists the following key points for TR24731:
29 - Guard against overflowing a buffer
30 - Do not produce unterminated strings
31 - Do not unexpectedly truncate strings
32 - Provide a library useful to existing code
33 - Preserve the null terminated string datatype
34 - Only require local edits to programs
35 - Library based solution
36 - Support compile-time checking
37 - Make failures obvious
38 - Zero buffers, null strings
39 - Runtime-constraint handler mechanism
40 - Support re-entrant code
41 - Consistent naming scheme
42 - Have a uniform pattern for the function parameters and return type
43 - Deference to existing technology
44 
45 and the following can be added...
46 
47 - provide a library of functions with like behavior
48 - provide a library of functions that promote and increase code safety and
49  security
50 - provide a library of functions that are efficient
51 
52 The C11 Standard adopted many of these points, and added some secure
53 _s variants in the Annex K. The Microsoft Windows/MINGW secure API did the
54 same, but deviated in some functions from the standard. Besides
55 Windows only Android's Bionic and Embarcadero implemented this C11
56 secure Annex K API so far.

59 Design Considerations
60 ---------------------
61 This library only implements a subset of the functions defined in the
62 specification. Included in the library are extensions to the specification to
63 provide a complementary set of functions with like behavior.

74 === Use of errno
75 
76 The TR24731 specification says an implementation may set errno for the
77 functions deﬁned in the technical report, but is not required to. This
78 library does not set errno. The library does use errno return codes as
79 required by functional APIs. Specific Safe C String and Safe C Memory errno
80 codes are defined in the safe_errno.h file.

83 === Runtime-constraints
84 
85 Per the spec, the library verifies that the calling program does not violate
86 the function's runtime-constraints. If a runtime-constraint is violated, the
87 library calls the currently registered runtime-constraint handler.
88 
89 Per the spec, multiple runtime-constraint violations in the same call to a
90 library function result in only one call to the runtime-constraint handler.
91 The first violation encountered invokes the runtime-constraint handler.
92 
93 The runtime-constraint handler might not return. If the handler does return,
94 the library function whose runtime-constraint was violated returns an
95 indication of failure as given by the function’s return.
96 
97 rsize_t::
98  The specification defines a new type. This type, rsize_t, is
99  conditionally defined in the safe_lib.h header file.
100 
101 RSIZE_MAX::
102  The specification defines the macro RSIZE_MAX which expands to a value
103  of type rsize_t. The specification uses RSIZE_MAX for both the string
104  functions and the memory functions. This implementation defines two
105  macros: RSIZE_MAX_STR and RSIZE_MAX_MEM. RSIZE_MAX_STR defines the
106  range limit for the safe string functions. RSIZE_MAX_MEM defines the
107  range limit for the safe memory functions. The point is that string
108  limits can and should be different from memory limits.

111 === Header Files
112 
113 The specification states the various functions would be added to existing
114 Standard C header files: stdio.h, string.h, etc. This implementation
115 separates the memory related functions into the safe_mem_lib.h header and the
116 string related functions into the safe_str_lib.h header.
117 You can optionally use the safe_lib.h header to include also non str/mem functions.
118 
119 The make file builds a single library libsafec-VERSION.a and .so in the
120 src/.libs directory.
121 Built but not installed are also libmemprims, libsafeccore and libstdunsafe.
122 
123 It is possible to split the make such that a separate safe_mem_lib.so and
124 safe_str_lib.so are built. It is also possible to integrate the prototypes
125 into the Standard C header files, but that may require changes to your
126 development tool chain.

129 Userspace Library
130 -----------------
131 The build system for the userspace library is the well known *GNU build
132 system*, a.k.a. Autotools. This system is well understood and supported
133 by many different platforms and distributions which should allow this
134 library to be built on a wide variety of platforms. See the
135 xref:tested-platforms[``Tested Platforms''] section for details on what
136 platforms this library was tested on during its development.

139 === Building
140 
141 For those familiar with autotools you can probably skip this part. For those
142 not and want to get right to building the code see below. And, for those that
143 need additional information see the 'INSTALL' file in the same directory.
144 
145 .To build you do the following:
146 ----
147  prompt$ ./build-tools/autogen.sh
148  prompt$ ./configure
149  prompt$ make
150 ----
151 'autogen.sh' only needs to be run if you are building from the git
152 repository. Optionally, you can do ``make check'' if you want to run the unit
153 tests.

166 Safe Linux Kernel Module
167 ------------------------
168 The build for the kernel module has not been integrated into the autotools
169 build infrastructure. Consequently, you have to run a different makefile to
170 build the kernel module.
171 
172 
173 === Building
174 
175 .To build do the following:
176 ----
177  prompt$ ./configure
178  prompt$ make -f Makefile.kernel
179 ----
180 
181 This assumes you are compiling on a Linux box and this makefile supports the
182 standard kernel build system infrastructure documented in:
183 <linux-kernel-src-tree>/Documentation/kbuild/modules.txt
184 
185 NOTE: If you build the kernel module then wish to build the userspace library
186  or vice versa you will need to do a +make clean+ otherwise a +make check+
187  will fail to build.
